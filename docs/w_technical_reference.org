* IRL at a Glance

IRL is a decentralized photo network that transforms digital media into verifiable Web3 assets. It uses blockchain storage, IPFS, NFTs, and the C2PA standard to create traceable provenance records for images and videos.

** Two-Contract Architecture (Custody NFT + CFS Verifier)

| Contract | Language | Purpose |
|----------+----------+---------|
| Custody NFT | Solidity | ERC-721 ownership token for registered assets |
| CFS Verifier | Rust/Stylus | On-chain Camera Flash Signature proof verification |

The CFS Verifier runs as a Stylus (Rust → WASM) contract on Arbitrum to minimize computational cost for cryptographic operations (10-100x cheaper than EVM for signature verification and hashing). The Custody NFT remains in Solidity to leverage OpenZeppelin's battle-tested ERC-721 implementation and marketplace compatibility.

*** Custody NFT Contract

*Standards*: ERC-721, EIP-2981 (royalties), OpenZeppelin AccessControl

*Purpose*: Represents exclusive ownership of a captured media asset. One NFT per asset, minted only when a valid CFS proof is verified on-chain. The holder controls licensing terms and receives royalties.

*Token ID derivation*: =uint256(keccak256(nid))= — deterministic from the IPFS CID. Given a file, anyone can compute its token ID. Aligns with the EIP-7053 "CID-as-identity" principle.

*Token URI*: Set at mint time, permanently immutable. Points to the original AssetTree on IPFS (=ipfs://{assetTreeCid}=). The NFT certifies origin; commit history lives in the Commit contract.

*Royalty*: EIP-2981, set at mint. Updatable by current NFT holder.

*Constraints*:
- One mint per Nid (reverts if token ID already exists)
- Non-burnable (custody is permanent; transfer is the only exit)
- No pause mechanism

*Interface*:

#+begin_src solidity
// Minting (MINTER_ROLE only — held by CFS Verifier contract)
mint(address to, string nid, string assetTreeCid, address royaltyReceiver, uint96 royaltyFee) → uint256 tokenId

// Holder actions
updateRoyalty(uint256 tokenId, address receiver, uint96 feeNumerator)

// Lookups
nidOf(uint256 tokenId) → string
tokenIdOf(string nid) → uint256
#+end_src

* Core Concepts

** Numbers ID (Nid)

The Nid is a 59-character string cryptographically generated from the file content. It is the IPFS Content Identifier (CID) and serves as the canonical Web3 address for digital media files. Follows the EIP-7053 standard for interoperable identification.

- Every file gets a unique Nid
- A file can only be registered once (deduplication by content hash)
- The Nid is the primary key for all queries

** Assetization Process

The workflow to convert a media file into a Web3 asset:

1. *Data Preparation*: Add metadata (creator, description, generation details) and create a digital signature. IRL Camera App automates this.
2. *Blockchain Registration*: Upload via API or IRL Camera App. System generates Nid, initial Commit, and AssetTree on-chain.
3. *Monitor and Govern*: AI-powered scanning and human-review for improper usage. Custody NFTs grant ownership certificates for policy definition.
4. *Monetization*: Create Asset Contracts for royalty distribution. Mint License NFTs tradeable on marketplaces.
5. *Commit and Trace*: Subsequent modifications tracked as Commits. Complete history traceable and verifiable.

** Commits

Commits are Git-like version control records for media files. They are immutable on-chain messages paired with IPFS-pinned AssetTree files.

*** Pre-Defined Actions

Initial registration (Arbitrum), NFT minting on various chains, and licensing. Developers can create custom actions by submitting JSON specifications to the Nit repository.

*** Key Properties

- Anyone can make commits for media files (decentralized contribution)
- Commits from owners, creators, and verified sources are distinguishable through filtering
- Single assets support multiple commits, building a history
- Original assets are NOT stored on-chain; only the Nid, hashes, and signatures are recorded

** AssetTree

AssetTree files are JSON documents stored on IPFS that describe all properties of a registered asset. Each commit generates a new AssetTree reflecting the asset's current state.

*** integrityCid: Device-Signed Proofs

The integrityCid field references tamper-proof metadata collected by the capture device:

- Device specifications (name, manufacturer, OS version, UUID)
- Environmental context (battery level, memory usage, disk space)
- Geolocation data (latitude/longitude)
- Proof hash: cryptographic fingerprint of content
- Capture timestamp and MIME type

NOTE: This is metadata-only integrity. There is no equivalent to IRL's Camera Flash Signature liveness proof. The device data can be spoofed on a compromised device.

* Comparative Analysis: IRL vs Number 

** Shared Architecture

Both projects use:
- IPFS/Filecoin for content storage
- Blockchain for integrity records
- C2PA standard for provenance metadata
- NFTs for ownership tracking
- Mobile camera app as primary capture tool
- Commit-based history model (Git-like)

** Key Differences

| Dimension | IRL | Numbers |
|-----------+------------------+---|
| Blockchain | Own L1 (Arbitrum) | Arbitrum (Ethereum L2) |
| Liveness proof | None (metadata only) | Camera Flash Signature |
| Token dependency | IRL required for every write operation | No token planned |
| Verification cost | 0.2 IRL per search | Free verification target |
| Primary positioning | Creator economy / monetization | Truth / journalism / censorship resistance |
| C2PA integration | Optional injection point | Embedded at capture with CFS |
| Security inheritance | Own validator set | Ethereum's security via Arbitrum |
| Device integrity | integrityCid (spoofable metadata) | CFS (cryptographic liveness proof) |
| Platform | Android + iOS | Android first |

** IRL's Competitive Advantages

1. *Camera Flash Signature*: The single strongest differentiator. Numbers has zero liveness proof. Their IRL Camera App collects device metadata (integrityCid) but cannot prove a real camera took the photo. CFS is IRL's moat.

2. *No token gate*: Numbers charges IRL for every operation. This creates friction for adoption. IRL can make basic verification free and charge only for premium features.

3. *Stronger security inheritance*: Arbitrum inherits Ethereum's validator security. Numbers runs its own L1 subnet with a smaller, less battle-tested validator set.

4. *Censorship resistance*: Numbers positions as a creator economy tool. IRL positions for journalism, human rights documentation, and anti-censorship. This is a fundamentally different and underserved market.

5. *C2PA at capture time*: Numbers treats C2PA as an import mechanism for files created elsewhere. IRL embeds C2PA manifests during capture with integrated CFS proof, creating a strictly stronger provenance chain.

** What IRL Should Adopt from Numbers

1. *Git-like commit model*: The Commit Contract + AssetTree pattern is proven and well-designed. IRL's CommitRegister should follow a similar structure.

2. *CID-as-identity (EIP-7053)*: Using the IPFS content hash as the canonical asset ID is elegant and standard. IRL should adopt this.

3. *Free reads, paid writes*: Reading commit history costs nothing in Numbers. This is the right model for maximizing verification reach.

4. *Nit-style CLI*: A developer-friendly abstraction over blockchain complexity. IRL should build equivalent tooling from the start.

5. *Verify Engine similarity search*: AI-powered near-duplicate detection adds value beyond exact-match lookups. IRL should include this.

6. *AssetTree schema*: The 21-field JSON metadata structure is well-thought-out. IRL should use a compatible schema extended with CFS-specific fields.

7. *IPTC digitalSourceType vocabulary*: Standard categorization of content origin (digital capture, AI-generated, etc.) is essential for IRL's use case.

** Risks and Gaps Revealed by Numbers

1. *Adoption despite investment*: Numbers has been live since 2020 with real partnerships (Reuters, elections) and notable investors, yet mainstream adoption remains unclear. IRL enters late but with stronger technology.

2. *Token economics as friction*: Every API call in Numbers costs IRL tokens. This creates a barrier for the journalism and human rights communities IRL targets.

3. *Metadata-only integrity is spoofable*: Numbers' integrityCid collects device metadata (battery level, GPS, OS version) but this data can be fabricated on a rooted device. CFS addresses this with a cryptographic proof that requires physical hardware interaction.

4. *Own L1 complexity*: Running a custom Arbitrum adds operational burden. IRL using Arbitrum is operationally simpler and benefits from a larger ecosystem.

* Design Decisions

** DD-001: Auto-mint Custody NFT on initial registration

*Decision*: The Custody NFT is minted automatically when a media file is first registered. One transaction covers both commit and ownership.

*Rationale*: Simplest UX, ensures every registered asset has a clear owner from the start. The CFS proof already validates authorship, so decoupling minting adds complexity without benefit.

*Alternatives rejected*: Separate mint transaction (extra step, no gain), claim window (unnecessary complexity for IRL's use case).

** DD-002: Camera app as sole minting authority via on-chain CFS verification

*Decision*: Only a valid Camera Flash Signature proof can trigger minting. The proof is verified on-chain by the CFS Verifier contract (Stylus/Rust), not by a backend relay.

*Rationale*:
- *Trustless*: No backend key to compromise. Verification is math on-chain.
- *Censorship resistant*: A relay can be shut down or pressured. An on-chain verifier cannot. Critical for journalism and human rights use cases.
- *Auditable*: Verification logic is public, immutable, and inspectable.

*Alternatives rejected*: Server-side relay (single point of failure, trust assumption), permissioned minter without CFS check (breaks authenticity guarantee).

** DD-003: Immutable token URI

*Decision*: Token URI is set at mint time and can never be updated. Points to the original AssetTree on IPFS.

*Rationale*: The Custody NFT's job is to certify origin — the moment of capture with CFS proof. If the URI can be swapped, the NFT no longer proves what it claims. Commit history (subsequent metadata, license changes, verification events) is readable from the Commit contract. The NFT points to the moment of truth and nothing else.

*Alternatives rejected*: Updatable URI (breaks provenance guarantee), dual URI with mutable "latest" pointer (unnecessary complexity, history belongs in Commit contract).

** DD-004: Nid-derived token IDs

*Decision*: Token ID is =uint256(keccak256(nid))= — deterministic from the IPFS content identifier.

*Rationale*: Aligns with EIP-7053 "CID-as-identity" principle. Anyone with the file can compute the token ID without querying the chain. No storage needed for mapping tables. Collision risk on keccak256 is negligible.

*Alternatives rejected*: Auto-increment (requires bidirectional mapping, loses content-addressable property).

** DD-005: Stylus for CFS verification, Solidity for ERC-721

*Decision*: Split architecture. CFS Verifier in Rust/Stylus, Custody NFT in Solidity.

*Rationale*: CFS verification involves cryptographic operations where Stylus is 10-100x cheaper than EVM. Solidity ERC-721 uses OpenZeppelin (battle-tested, marketplace-compatible). Best of both worlds.

*Alternatives rejected*: Unified Stylus contract (would reimplement ERC-721 in Rust with less audited tooling), pure Solidity (prohibitively expensive for crypto verification on-chain).

** DD-006: EIP-2981 for royalties

*Decision*: Standard EIP-2981 royalty interface. Single recipient, updatable by NFT holder.

*Rationale*: Widely supported by marketplaces (OpenSea, Blur). Simple. If multi-party splits are needed later, the royalty recipient can be set to a splitter contract.

*Alternatives rejected*: Custom on-chain royalty splitting (premature complexity), no royalties (limits monetization from day one).
